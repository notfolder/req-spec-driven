# ヒアリング時の複雑さ評価ルール

ヒアリングで選択肢を提示する際に、MVPから逸脱しないための共通ルールを定義する。

## 基本原則

- 選択肢を提示する場合は、全ての選択肢に複雑さを定量表示する。
- 推奨は必ず「業務課題を満たす中で複雑さが最小の選択肢」とする。
- ユーザーが高複雑な選択肢を選ぶ場合は、必須理由の再確認を行う。

## 複雑さスコアの算定

### 評価軸（各 0-2 点）

- データ変更量: スキーマ変更、移行、整合性考慮の有無
- 画面/操作追加量: 新規画面数、操作分岐数、説明負荷
- 外部連携負荷: 連携先数、認証、失敗時ハンドリング
- 非機能制約負荷: 応答時間、可用性、セキュリティ要件の厳しさ
- 運用変更負荷: 監視、アラート、手順追加、教育コスト

### スコア計算

- 合計点 T = 5軸の合計（0-10 点）
- 複雑さレベル C（1-5）は以下で決定する
  - C=1: T が 0-1
  - C=2: T が 2-3
  - C=3: T が 4-6
  - C=4: T が 7-8
  - C=5: T が 9-10

## 質問時の提示ルール

- 選択肢ごとに、以下を必ず併記する
  - 複雑さ（C=1-5）
  - 根拠（どの評価軸が効いているか）
  - MVP適合性（適合/条件付き/不適合）
- 推奨は1つに限定し、推奨理由を一文で明示する。

## 高複雑さ選択時の再確認ゲート

- ユーザーが推奨以外、または C がより高い選択肢を選んだ場合は、次の確認を必須とする。
  - 最小案では満たせない業務上の必須理由は何か
- 必須理由が具体化できない場合は、最小案を再提示し、最小案で確定する。

## 同点時の優先順位

同じ複雑さレベルの場合は、以下の順で小さいものを優先する。

1. 外部連携が少ない
2. データ変更が少ない
3. 画面/操作追加が少ない
4. 運用変更が少ない

## 禁止事項

- 複雑さを示さずに選択肢を提示することを禁止する。
- 推奨理由を示さずに選択肢を提示することを禁止する。
- 高複雑さ選択時の再確認ゲートを省略することを禁止する。
