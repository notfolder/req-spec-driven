# 詳細設計書の必須章立て

以下の全項目を必ず含める。
各セクションに登場する全ての設計要素（クラス・API・関数・データモデル等）には、必ず `isdd-common/references/id-definitions.md` に従った `DS-*` IDを付与すること。IDのない設計要素は記述しない。

### 1. 言語・フレームワーク

- 言語・フレームワークは選定理由を必ず明記する
- 特段の理由がなければ、言語は **Python** を選定する

**GUI が必要な場合の選定ルール**

| 選択肢 | 適用条件 | 備考 |
|---|---|---|
| **Streamlit** | 基本・推奨 | シンプルなUI、単一ページアプリ向け |
| **Vue（Vuetify）+ FastAPI** | 複雑な画面遷移や部品が必要 | マルチページ、複雑な状態管理が必要な場合 |

**Vue + FastAPI を選ぶ場合の必須設計記載**

- フロントエンド：マルチステージビルド後、Nginx で配信する
- バックエンド：Nginx でリバースプロキシを設定する
- API エンドポイント：`/api/` を前提に統一する

### 2. システム構成

- コンポーネント一覧（要件を根拠に表形式）
- システム全体構成図（mermaid）
- 各コンポーネントの役割と機能
- コンポーネント間インターフェースとデータフロー
- ネットワーク構成図（mermaid）

### 3. データベース設計

- DB の必須性を判定し、理由を必ず明記する
- DB が必須でない場合は「DB は必須ではないため、DB 設計は行わない」と明記する

**DB 製品の選定ルール**

| データ規模・複雑度 | 選択肢 | 備考 |
|---|---|---|
| 極端に少量（目安 100 万件以内）かつ単純 | **SQLite** | ファイルベース、セットアップが簡単 |
| それ以外で DB 必須 | **PostgreSQL** | 本格的なRDB、多機能 |

**DB 設計に含める項目**

- テーブル設計：カラム、型、制約を明記
- リレーション図：mermaid 形式で描画

### 4. アーキテクチャ設計

- 外部設計
  - UI設計（GUIなら画面一覧、画面遷移図、AAモックアップ。CUIならコマンド仕様）
    - 画面一覧：各画面の要素と機能を説明
    - CUI：コマンドや引数、設定の一覧を表にする
  - 外部システム連携（一覧、インターフェース、データフロー図）
    - 要件に外部連携がある場合、`external/[システム名]` を参照して設計を行うこと。
    - 外部連携先のシステムを呼び出す際には、`external/[システム名]/src` を使って呼び出す設計にすること。
    - 外部連携先のシステムを直接呼び出す設計は禁止する。
  - 外部DB連携（一覧、インターフェース、データフロー図）
- 内部設計
  - 処理フロー図（mermaid）
  - 各処理の役割と機能
  - 必要に応じたバッチ設計（フロー図、役割）

### 5. クラス設計

クラス設計にあたってはSOLID原則を遵守すること。

- 全クラス一覧と役割
  - SOLID原則への適合状況を表で明示
- クラスの責務、主要属性、主要メソッド
- クラス図（mermaid）
- システム内メッセージ一覧と役割
- メッセージフロー図（mermaid）

### 6. その他設計

- エラーハンドリング設計（想定エラー一覧）
- セキュリティ設計

### 7. コード設計

- ソースコードのディレクトリ構成（AA）：ファイル名を含めないこと
- 各ディレクトリ配下のファイル名、役割、含まれるクラスの一覧表：各行に関係する設計IDを明記すること
- 言語・フレームワークに応じたコーディング規約表

### 8. テスト設計

- テスト種別と内容
- 各テストの目的と方法
- 実装すべき全テストケース
  - 全モジュールの単体テスト
  - 全機能要件の結合テスト
  - システム全体の総合テスト
  - GUI/CUIがある場合のユーザー視点テスト（e2eテスト）

### 9. 運用設計

- 基本起動方式はdocker compose
- DBスキーマや初期ユーザー等の初期化が必要なら起動時自動化
- 起動方法と操作説明をREADME.mdに記載する旨を設計書に明記

### 10. ログ・監視・アラート設計

- ログ設計の必須性判定
  - 必須でない場合は「ログの設計は必須ではないため、ログの種類と内容の記述は行わない」と明記
- 監視・アラート設計の必須性判定
  - 必須でない場合は「監視・アラートの設計は必須ではないため、監視・アラートの内容と対応方法の記述は行わない」と明記
- 必須の場合は内容と対応方法を表で定義

### 11. E2Eテスト設計

**設計要件**

- 要件書のテスト用利用シナリオを 100% 網羅する
- 各シナリオごとに「目的、前提条件、手順、期待結果」を表で定義する
- GUI の場合は全画面遷移を網羅し、Playwright 前提で具体化する
- 外部連携がある場合、E2E は `mock` モードと `real` モードの2モードを設計する
- 設計上は全 E2E シナリオを `mock` / `real` の両モードで実行対象として定義する

**docker compose への test_playwright サービス追加**

- `mcr.microsoft.com/playwright:v1.59.0` を使用する
- プロファイル名は `test` に加えて、外部連携時は `mock` / `real` の切替プロファイルを定義する
- 通常起動では起動しない
- テストコードをマウントして即時反映できる構成にする

**E2E実行コマンドの設計**

設計に E2E 実行コマンドを明記する。実行コマンドは以下の形式を前提とする。

```
docker compose run --rm test_playwright sh -c "npm install && npx playwright test"
```

**E2E運用設計**

以下の項目を設計に含める。

- ファイル配置：プロジェクトルート `e2e` 配下にテスト資産を配置する
- 外部モックは `external/[システム名]/mock/` を正本とし、`e2e` 側はシンボリックリンクで参照する（Git 管理下に置く）
- モード切替は docker compose プロファイルとテスト実行コマンドの両方を提供する
- テスト環境作成スクリプトは原則 bash で実装し、複雑処理はメインプロジェクト言語または Python を利用する
- テスト実行時の URL は compose 内サービス名ベースで定義する
- 実装時・CI 時の必須ゴールは `mock` モード E2E 全件通過とする
- `real` モード E2E は手動実行のみ必須とする
- テスト失敗時は修正と再実行を繰り返し、全成功まで継続する方針を明記する
