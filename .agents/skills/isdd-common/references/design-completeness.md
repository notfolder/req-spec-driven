# isdd 設計完全性制約

isdd 詳細設計スキル（isdd-design、isdd-change-design）で作成した設計書の完全性を検証するためのチェックリスト。

---

## 完全性チェック項目

以下を満たさない場合は不足として **必ず指摘して修正する**。

### エンティティ・データ整合性

- 業務エンティティ一覧と、各エンティティに対する CRUD、一覧、詳細、検索、状態管理の定義が完全であるか
- エンティティと画面、API、クラスの対応関係が明確であるか
- データ整合性（PK、FK、業務制約）が定義されているか

### トランザクション・排他制御

- トランザクション境界とロールバック条件が定義されているか
- 排他制御（楽観または悲観）が選択・設計されているか

### API 設計

- API 入出力、バリデーション、エラー仕様が定義されているか

### セキュリティ・監査

- 認証、認可、監査ログが設計されているか

### 運用・監視

- ログ、監視、アラート、障害対応が定義されているか

### 状態遷移・ビジネスロジック

- 各エンティティの状態遷移が定義されているか

### テスト

- 全機能に対応する正常系、異常系テストが設計されているか

### コード品質

- 同一意味処理の重複実装が排除され、再利用方針が明記されているか

### 要素の削除

- 削除可能な設計要素を列挙しているか
- 削除しても要件を満たす要素は実際に削除しているか

### ID 付与

- 全ての設計要素に `DS-*` ID が付与されているか（ID なし要素がある場合は不足として修正する）

---

## 外部連携時の確認

- 要件に外部連携があるのに、`external/[システム名]` の内容が完全でない場合は、不足としてユーザーに指摘する
